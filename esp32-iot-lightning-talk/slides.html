Same Arduino IDE / framework — but built for IoT from day one.

Reading a sensor BME280 over I²C


#include <Wire.h>
#include <Adafruit_BME280.h>

Adafruit_BME280 bme;

void setup() {
  Serial.begin(115200);
  Wire.begin(21, 22);            // SDA, SCL on ESP32
  bme.begin(0x76);
}

void loop() {
  float t = bme.readTemperature();
  float h = bme.readHumidity();
  float p = bme.readPressure() / 100.0f;
  Serial.printf("t=%.2f h=%.2f p=%.2f\n", t, h, p);
  delay(10000);
}

Three lines do what used to need a datasheet and a soldering iron.

From sensor to InfluxDB

Time-series DB. Wire format is dead simple: line protocol.


weather,room=office,sensor=bme280 temp=21.4,hum=43.1,press=1013.2
<measurement>,<tags>            <fields>                         [<ts>]
  • tags = indexed labels (room, sensor, fw_version)
  • fields = the actual numbers
  • Push over POST /api/v2/write — done.

The Arduino client makes it one-liner-ish


#include <WiFi.h>
#include <InfluxDbClient.h>       // influxdata/influxdb-client-arduino

InfluxDBClient db(INFLUX_URL, INFLUX_ORG, INFLUX_BUCKET, INFLUX_TOKEN);
Point          weather("weather");

void setup() {
  WiFi.begin(SSID, PASS);
  while (WiFi.status() != WL_CONNECTED) delay(200);
  weather.addTag("room",   "office");
  weather.addTag("sensor", "bme280");
}

void loop() {
  weather.clearFields();
  weather.addField("temp",  bme.readTemperature());
  weather.addField("hum",   bme.readHumidity());
  weather.addField("press", bme.readPressure() / 100.0f);
  db.writePoint(weather);
  delay(10000);
}

Over-the-air updates

…because that sensor is now glued behind your fridge.

  • Bug in your sleep logic? You're not unscrewing 12 boxes.
  • New sensor field? Roll forward, no field trip.
  • Security fix in arduino-esp32? Same story.

ESP32's flash layout is designed for this from the start.

How it works: two app partitions


 ┌──────────┬──────────┬──────────┬──────────┬──────────┐
 │ bootldr  │ nvs/data │  ota_0   │  ota_1   │  spiffs  │
 │  +otadata│          │  (app A) │  (app B) │  (fs)    │
 └──────────┴──────────┴──────────┴──────────┴──────────┘
                              ▲                ▲
                              │                │
                       running now      next update lands here
  • Bootloader reads otadata → boots whichever slot is "active".
  • New firmware → write to the other slot, then flip the pointer.
  • If the new image fails to boot / call esp_ota_mark_app_valid_cancel_rollback → bootloader rolls back. Atomic, safe.

Quick & local: ArduinoOTA


#include <ArduinoOTA.h>

void setup() {
  WiFi.begin(SSID, PASS);
  ArduinoOTA.setHostname("sensor-office");
  ArduinoOTA.setPassword("hunter2");
  ArduinoOTA.begin();
}

void loop() {
  ArduinoOTA.handle();          // ← that's it
  // … your sensor code …
}

mDNS-discovered from the Arduino IDE / arduino-cli / espota.py. Great for a dev bench, not for fleets.

Production pattern: HTTPS pull + rollback


#include <HTTPUpdate.h>

void checkForUpdate() {
  WiFiClientSecure client;
  client.setCACert(ROOT_CA);
  httpUpdate.rebootOnUpdate(true);

  // Server returns 200 + new .bin, or 304 if we're current
  auto r = httpUpdate.update(
    client,
    "https://fw.example.com/sensor/latest.bin",
    FW_VERSION);                     // sent as User-Agent / header

  if (r == HTTP_UPDATE_FAILED)
    log_e("OTA failed: %s", httpUpdate.getLastErrorString().c_str());
}
  • Sign & pin the cert. Sign the firmware too (secure boot v2).
  • Mark valid only after a self-test → automatic rollback if you brick it.
  • Fleet tooling: ESPHome, Toit, Mender, or a CI job + S3.

tl;dr

  • ESP32 = an Arduino that ships with WiFi, deep sleep, and a real CPU.
  • InfluxDB line protocol + the Arduino client → ~20 lines of code.
  • OTA isn't optional once the box is glued behind the fridge.
  • Two partitions + rollback = sleep-at-night firmware updates.
