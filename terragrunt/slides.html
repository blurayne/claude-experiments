terragrunt

what Terraform forgot, and how 1.0 finally finished it

10-minute lightning talk · space to advance · s speaker notes

Audience: people who already write Terraform. No "what is IaC" intro. Promise: in 10 minutes you'll know what Terragrunt adds, what units/stacks are, what 1.0 changed, and how it compares to Atmos.

Where vanilla Terraform hurts at scale

  • The Terralith: one root module, one state — every plan is slow, every apply risks everything, one lock for the whole team.
  • The backend block forbids variables → backend config is copy-pasted into every environment.
  • Provider blocks, versions, tags: duplicated per root module.
  • Split the state and you get the next problem: no orchestrator. Nothing in Terraform runs 30 states in dependency order.
  • Promoting dev → staging → prod = copy-paste directories and pray the drift is intentional.
Key line: Terraform is great at ONE state. It has no story for MANY states. terraform_remote_state is read-only glue, not orchestration.

"But Terraform has workspaces…" advice

Workspaces wanted to solve

Multiple states for one configuration — cheap parallel copies (feature branches, spikes, ephemeral test envs).

  • Same backend, same credentials, same code version for every workspace
  • Active workspace = invisible CLI state → "applied to prod by accident"
  • terraform.workspace conditionals scatter env logic through the code
  • Even HashiCorp's docs say: not for strong dev/prod isolation

Where -var-file was the right call


terraform init  -backend-config=envs/prod.backend.hcl
terraform apply -var-file=envs/prod.tfvars
  • Envs differ only in values — same structure, same version, one account
  • Small estate, one team → simplest thing that works
  • Breaks down when envs need different module versions, accounts/roles, or shape

A unit is essentially "tfvars + backend-config + version pin" made first-class and per-directory — Terragrunt passes inputs as TF_VAR_* under the hood.

Advice framing: workspaces are fine for ephemeral copies of the SAME thing. The moment envs drift in version, account or shape, you're fighting the tool. tfvars + partial backend config is the honest small-scale path — Terragrunt is that pattern, industrialized.

Wrappers all the way down

terragrunt
↓ wraps (exec + generated files)
tofu / terraform — one single binary
↓ wraps (go-plugin · gRPC)
terraform-provider-aws …-google …-kubernetes
  • Terraform itself is an orchestrator of single binaries — every provider is a separate plugin binary spoken to over RPC.
  • Terragrunt applies the same trick one level up: a thin single binary that drives the real tofu/terraform.
  • It reimplements nothing — every plan/apply is the genuine article. TG_TF_PATH picks the wrapped binary.
The analogy the audience should keep: providers are plugins to Terraform; Terraform is a "plugin" to Terragrunt. Thin wrapper = low lock-in, easy to remove.

Under the hood: one terragrunt apply

  1. Parse & merge config — read terragrunt.hcl, walk up the tree for includes, evaluate HCL functions (find_in_parent_folders(), path_relative_to_include(), …).
  2. Resolve dependency blocks — run tofu output -json in each upstream unit (or substitute mock_outputs).
  3. Fetch the module source — via go-getter (same library Terraform uses) into .terragrunt-cache/<hash>/.
  4. Generate files — write backend.tf, provider.tf, … into that working dir.
  1. Prepare the environment — inputs become TF_VAR_* env vars; assume the unit's iam_role if set.
  2. Auto-init if needed (and bootstrap the backend if asked).
  3. Run hooks & exec — before_hooks, then fork the real tofu/terraform as a subprocess, stream its output, retry on known transient errors, after_hooks / error_hooks.

run --all = do the above per unit, scheduled over the dependency DAG with a parallel worker queue.

No daemon, no state of its own, no lock-in: delete terragrunt.hcl + .terragrunt-cache/ and you're back to plain Terraform.

Demystify: Terragrunt is a config preprocessor + process supervisor. Everything it "does" is either a generated file, an env var, or a subprocess call. .terragrunt-cache is disposable; the state stays wherever your backend is. Debug tip: `terragrunt render --json` shows the fully-evaluated config, and the cache dir contains exactly what terraform actually saw.

The core concept: units vs modules

Terraform moduleTerragrunt unit
What it isreusable .tf codea directory with terragrunt.hcl
Roleblueprintdeployed instance of a module
Statenone of its ownits own state, always
Containsresources, logicmodule ref + version + inputs, no logic
Blast radiuswherever it's usedjust this unit

Terraform answers "how is a VPC built?" — a unit answers "which VPC, in which account, at which version, with which inputs?"

Modules were never the problem — instantiating them per env/account/region was. A unit is the missing noun: the *deployment* of a module, with identity and state.

A unit is ~15 lines


# live/prod/eu-central-1/vpc/terragrunt.hcl
include "root" {
  path = find_in_parent_folders("root.hcl")   # inherit backend & provider
}

terraform {
  source = "git::github.com/acme/modules.git//vpc?ref=v1.4.0"
}

inputs = {
  cidr = "10.32.0.0/16"
  azs  = 3
}

Pin a version per environment: dev runs v1.5.0-rc1, prod stays on v1.4.0. Promotion = bump one ref.

Point at the ref: this is how environments differ without copy-pasting code — same module, different version + inputs. The unit file is boring on purpose.

One backend definition, N states


# live/root.hcl — written ONCE for the whole estate
remote_state {
  backend = "s3"
  generate = { path = "backend.tf", if_exists = "overwrite" }
  config = {
    bucket = "acme-tf-state"
    key    = "${path_relative_to_include()}/tofu.tfstate"  # ← per-unit key
    region = "eu-central-1"
    use_lockfile = true
  }
}

generate "provider" {
  path     = "provider.tf"
  if_exists = "overwrite"
  contents = <<EOF
provider "aws" { region = "eu-central-1", default_tags { ... } }
EOF
}

Terragrunt generates the files Terraform refuses to parameterize. The directory path becomes the state key.

This was the original 2016 reason Terragrunt exists: backend blocks can't interpolate. Solution: don't fight it — generate the file before calling terraform.

Wiring units: dependency


# live/prod/eu-central-1/eks/terragrunt.hcl
dependency "vpc" {
  config_path = "../vpc"

  mock_outputs = {                 # lets `plan` work before vpc exists
    vpc_id = "vpc-mock0000"
  }
}

inputs = {
  vpc_id = dependency.vpc.outputs.vpc_id   # output → input, across states
}
  • Outputs of one state become inputs of another — explicitly, versioned, greppable.
  • All dependency edges together form a DAG over your whole estate.
mock_outputs matter in CI: plan a whole new environment before anything exists. The DAG is declared, not inferred — you can `terragrunt dag graph` it.

The missing orchestrator: run --all


$ terragrunt run --all plan          # every unit, in DAG order, in parallel
$ terragrunt run --all apply --filter ./prod/eu-central-1/...
$ terragrunt run --all apply --filter '--changed'   # only units touched in git
vpc → eks → app   ·   dns → cdn   (independent chains run in parallel)
  • Applies in dependency order, destroys in reverse order.
  • Failures stop the downstream part of the graph, not unrelated chains.
  • Plus quality-of-life: auto-init, retries on transient errors, before/after hooks.
This is the answer to "we split our Terralith into 40 states and now deploys are a runbook". One command walks the graph.

The problem Terragrunt created

Hundreds of tiny terragrunt.hcl files,
90% identical.

  • 3 environments × 4 regions × 12 components = 144 near-clones.
  • Community workarounds everywhere: _envcommon/, symlinks, custom generators.
  • DRY code, but WET structure.

This is exactly what Stacks — the headline of 1.0 — fix.

Honest slide: Terragrunt solved state-splitting and orchestration but paid with file sprawl. Stacks are the official answer to a decade of _envcommon hacks.

Stacks: units, generated 1.0


# live/prod/eu-central-1/terragrunt.stack.hcl
unit "vpc" {
  source = "git::github.com/acme/units.git//vpc?ref=v2.1.0"
  path   = "vpc"
  values = { cidr = "10.32.0.0/16" }
}

unit "eks" {
  source = "git::github.com/acme/units.git//eks?ref=v2.1.0"
  path   = "eks"
  values = { cluster_name = "prod-eu" }
}

stack "observability" {                    # stacks compose stacks
  source = "git::github.com/acme/stacks.git//o11y?ref=v0.9.0"
  path   = "o11y"
}
  • terragrunt stack generate materializes the units under .terragrunt-stack/ — run --all just works on them.
  • One versioned file describes a whole environment; values flow into each unit.
Mental model: unit = instance of a module; stack = instance of a *pattern* of units. Environments shrink from 40 files to one stack file. Stacks nest.

Terragrunt 1.0 2026-03-30

  • After ~10 years and 900+ releases of v0.x — it's finally, officially done.
  • Stacks GA — terragrunt.stack.hcl as the standard way to define estates.
  • Backwards-compatibility guarantee: semver for real; breaking changes only in rare majors. v0.x could break any Friday.
  • CLI redesign:
    • run-all → terragrunt run --all, flags lose their --terragrunt- prefix
    • new commands: exec · find · list · backend
    • 7 targeting flags collapsed into one --filter (paths, deps, git-changed)
  • First-class OpenTofu: tofu is the preferred wrapped binary when installed.
The compat guarantee is the real 1.0 feature — teams can finally upgrade without reading 40 changelog entries. Ten years from "hack around the backend block" to a stable product.

Secrets & backend init, compared

Terraform / OpenTofuwith Terragrunt
Secrets in TF_VAR_* env vars, Vault/SSM/Secrets-Manager data sources, sensitive flag (masks logs only), ephemeral values (≥ 1.10), OpenTofu state encryption sops_decrypt_file("secrets.yaml") → encrypted secrets live in git, next to the unit (KMS/age/GPG); get_env(); per-unit iam_role / --auth-provider-cmd → different credentials per env
Caveat both: values written to a resource end up in state in plaintext — sensitive ≠ encryption; encrypt the state at rest (or OpenTofu client-side)
Backend config static backend block, no variables; DRY only via init -backend-config=… per env one remote_state in root.hcl, generated per unit, path ⇒ state key
Backend exists? chicken & egg: bucket/lock table must pre-exist (bootstrap stack or clickops) terragrunt backend bootstrap creates + configures them (versioning, encryption, locking); backend migrate moves state
init manual, per root module, easy to forget after changes auto-init before every run
Two honest points: Terragrunt does NOT fix the state-plaintext problem — nothing wrapping terraform can. It fixes the workflow around secrets (SOPS in git, per-unit roles). And it kills the backend chicken-and-egg: bootstrap is a command, not a wiki page.

And Atmos? quick compare

TerragruntAtmos (Cloud Posse)
Config languageHCL (like Terraform)YAML stack manifests
Building blockunit → modulecomponent (a TF root module)
Environment modelfilesystem hierarchy + includefilesystem is just folders; imports + deep-merge define everything
Reuse mechanisminheritance up the dir treeYAML inheritance, mixins, catalogs
Extra scopetofu/terraform onlyalso helmfile, workflows, vendoring, validation (OPA/JSON-Schema)
Feelthin wrapper, stays close to TFfull framework + conventions (tenant/env/stage)

Rule of thumb: incremental adoption & HCL-native → Terragrunt. Big multi-account estate, config-as-data, team willing to adopt a framework → Atmos.

Same disease, different cure: Terragrunt keeps you in HCL and the directory tree IS the model. Atmos moves all semantics into YAML — folders mean nothing, imports and deep-merging mean everything. Atmos is more powerful and more opinionated; Terragrunt is easier to add to an existing repo (and to remove again).

tl;dr

  • Terraform scales resources; Terragrunt scales states.
  • Unit = a module instance with its own state. Stack = a generated pattern of units.
  • DRY backend/provider, explicit dependency DAG, run --all orchestration.
  • 1.0 (2026): stacks GA, redesigned CLI, and — after 10 years — a compat promise.
  • It's wrappers all the way down: providers ← terraform/tofu ← terragrunt.
